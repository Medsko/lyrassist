# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Lyrassist is a web app that assists a human lyricist — it sparks songs, it doesn't write them. The home screen is a grid of "assist modes" (Word Sparks, Object Writing, more planned in `roadmap.md`). Two-part monorepo: Kotlin/Spring Boot 4 API in `backend/`, React 19 + TypeScript + Vite + React-Bootstrap frontend in `frontend/`, PostgreSQL 17 in a container.

## Commands

Database (required by both the backend and the backend build):

```sh
podman start lyrassist-postgres        # container already exists on this machine
# podman compose up -d fails here (no compose provider); see README for the initial podman run
```

Backend (from `backend/`):

```sh
./gradlew bootRun    # starts on :8080; first run applies Flyway migrations and seeds ~15k words
./gradlew build      # includes a @SpringBootTest context test that NEEDS Postgres running
./gradlew test       # same caveat; single test: ./gradlew test --tests 'nl.medsko.lyrassist.LyrassistApplicationTests'
```

Frontend (from `frontend/`):

```sh
npm run dev          # Vite on :5173, proxies /api to :8080
npm run build        # tsc -b && vite build — use this as the type check
npm run lint         # oxlint
```

There are no frontend unit tests. For end-to-end verification (launching everything and driving the API/browser), use the project's `verify` skill — it has the launch steps, curl probes, Playwright selectors, and reseed procedure.

## Architecture

**Backend** (`backend/src/main/kotlin/nl/medsko/lyrassist/`) is a conventional layered Spring app: `controller` → `service` → `repository` (Spring Data JPA) over entities in `domain`. Request/response records live in `dto/Dtos.kt`. Each assist mode gets its own controller/service pair under `/api/<mode>` (see README for the endpoint table). Validation errors surface as RFC-7807 problem details; the frontend reads `body.detail`.

- **Schema is Flyway-owned** (`resources/db/migration/`, `ddl-auto: validate`): any entity change needs a new `V<n>__*.sql` migration.
- **Dictionary**: `word` table holds WordNet-derived adjectives + nouns (lemma, part_of_speech, pronunciation, syllable_count). `DictionarySeeder` (an `ApplicationRunner`) fully seeds when the table is empty and backfills pronunciations in place on databases seeded before they existed. The CSVs (`lemma,pronunciation,syllable_count`) are generated by the `buildDictionary` Gradle task (`tools/BuildDictionary.kt`, input paths via `-P` properties) from WordNet 3.1 + Norvig's frequency list + the CMU Pronouncing Dictionary; after regenerating, truncate `word` (cascades) and restart to reseed.
- **Rhyme logic** lives in `rhyme/RhymeClassifier.kt` — pure ARPAbet phoneme classification into Pattison's rhyme spectrum, unit-tested without Spring (`./gradlew test --tests 'nl.medsko.lyrassist.rhyme.RhymeClassifierTest'` needs no database).
- Entities are `class`es kept open via the Gradle `allOpen` block; the `word` table is shared by all modes via FK.

**Frontend** (`frontend/src/`):

- `main.tsx` defines the React Router tree: `App.tsx` is the layout shell (navbar + `<Outlet/>`), one page component per mode under `pages/`.
- `modes.ts` is the home-grid registry — a new mode means an entry here (name/description/path), a page in `pages/`, and a route in `main.tsx`. Modes without a `path` render as "coming soon" tiles.
- `api.ts` is the single hand-written API client: typed wrappers over `fetch` that throw `ApiError` (carrying HTTP status + problem-detail message). Add new endpoints here, not inline in components.
- `notepad/NotepadContext.tsx` is the one app-wide lyrics notepad (provider above the router, panel rendered by `App.tsx`): the draft survives route changes, buffers every keystroke to localStorage, and the explicit Save button/Ctrl+S persists it as a backend *snippet* (`/api/snippets` — reusable free text at any granularity; future modes like cut-up mine these). `append()` is the hook for modes to push text into it.
- `timer/TimerContext.tsx` is the one app-wide timer (provider above the router, widget in the navbar). It survives route changes; Object Writing drives it via `useTimer()`. Don't add per-page timers.

## Gotchas

- Gradle auto-provisions a JDK 21 toolchain via foojay (system Java is 25); first build downloads it.
- Spring Boot 4: starter names differ from Boot 3 (e.g. `spring-boot-starter-webmvc`, per-starter test artifacts) and Jackson is `tools.jackson.*`.
- DB credentials are lyrassist/lyrassist/lyrassist on localhost:5432 (see `application.yaml`, `compose.yaml`).
